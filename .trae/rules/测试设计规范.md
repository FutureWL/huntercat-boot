# 测试设计规范

## 1. 目标

- 保障核心业务逻辑在重构与迭代中可回归验证
- 让缺陷尽量在本地与 CI 阶段被发现，而不是在生产环境
- 通过可重复、可读、可维护的测试用例降低长期技术债

## 2. 分层策略

### 2.1 单元测试（Unit）

- 范围：纯函数、工具方法、业务规则计算、数据转换、校验逻辑
- 原则：不依赖真实网络、真实数据库、真实外部服务；通过 Mock/Stub 隔离依赖
- 关注点：输入输出、边界条件、异常分支、幂等与一致性（如同一输入多次调用结果一致）

### 2.2 集成测试（Integration）

- 范围：模块间协作（例如 repository + service）、API 路由与中间件链路
- 原则：允许使用真实依赖的“测试实例”（如本地/容器 DB、Redis），但必须可一键启动/一键清理
- 关注点：契约一致性、数据持久化、事务/幂等、鉴权与权限、缓存一致性

### 2.3 端到端测试（E2E）

- 范围：用户关键路径（注册/登录/退出、核心 CRUD、关键页面跳转）
- 原则：尽量贴近真实用户行为；优先覆盖最关键的 20% 路径
- 关注点：可访问性（基础 aria）、跨页面状态、错误提示、浏览器兼容性（按项目要求选择）

## 3. 用例设计要求

- 必须覆盖：正常路径、边界值、异常路径（输入非法、权限不足、资源不存在、外部依赖失败）
- 重点覆盖：鉴权（未登录/已登录/会话过期）、幂等（重复提交）、并发（同一资源多次写入）
- 对不稳定点：时间、随机数、外部 IO 必须固定化（例如注入 clock、mock random）
- 对错误处理：断言错误码/错误信息/错误类型，而不是只断言“抛错了”

## 4. 命名与结构

- 用例命名：描述“行为 + 条件 + 期望结果”，避免“test1/test2”
- 目录结构：按业务域或模块组织测试文件，避免把所有测试堆到一个目录
- 测试数据：集中管理测试 fixture，避免在多个测试中复制粘贴大段数据

## 5. Mock 与隔离

- Mock 只 Mock 边界：外部服务、网络请求、存储层适配器、时间与随机数
- 尽量不要 Mock 被测模块内部实现细节（例如私有函数），优先验证可观察的行为
- 对共享 Mock：必须可复用且默认安全（不污染其他用例），用例结束后恢复（reset/restore）

## 6. 数据与环境

- 测试环境必须与开发环境隔离：使用独立的数据库/Redis 前缀/对象存储桶
- 测试前置与清理：必须可重复执行；清理失败不得影响下一次运行
- CI 中不得依赖手工步骤：所有依赖必须自动启动或可被替代（如 mock server）

## 7. 质量门槛（建议）

- 核心业务模块必须有测试覆盖；公共工具函数优先补齐单元测试
- 失败必须可定位：断言信息清晰，必要时输出最小必要上下文（禁止输出密钥与敏感信息）
- 避免脆弱测试：不要依赖 DOM 结构细枝末节、不要依赖不稳定的时间/动画

## 8. 约束

- 禁止在测试与日志中写入真实密钥、token、内网地址、个人敏感信息
- 测试代码同样遵循项目编码规范与安全规范
